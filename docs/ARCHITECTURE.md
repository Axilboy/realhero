# Архитектура Real Hero (веб → нативные клиенты)

## Цель

Один **бэкенд (API)** и одна **БД** обслуживают:

| Клиент | Роль |
|--------|------|
| **Веб** (корень репозитория, Vite + React) | Браузер, Telegram Mini App (WebView) |
| **Android / iOS (будущее)** | Оболочка (Capacitor, React Native и т.д.) с тем же UI или отдельным UI — данные только через API |

Бизнес-логика **не дублируется** в клиентах: только отображение и UX.

## Поток данных

```text
[ Web SPA ] ──┐
[ TG Mini ] ──┼── HTTPS ──► [ API : Fastify ] ──► [ PostgreSQL / SQLite ]
[ Android ] ──┤              /api/v1/...
[ iOS     ] ──┘
```

## Репозиторий

- Фронтенд — в корне репозитория (текущая структура).
- **`api/`** — бэкенд (Fastify + Prisma), в **том же** git-репозитории.

При росте проекта при необходимости вынесите `api` в `packages/api` (monorepo) — контракт тот же.

## Контракт API

- Префикс: **`/api/v1/`** (версионируемый).
- Аутентификация (план): **JWT** или **сессия** для веба; для Telegram — проверка **`initData`** на сервере + привязка к `User.telegramId`.
- Документация (план): **OpenAPI** (`/api/v1/openapi.json`) и при необходимости генерация клиентов для TypeScript / Kotlin / Swift.

## CORS

Список разрешённых origin задаётся в **`CORS_ORIGINS`** (через запятую): локальная разработка, прод-домен, при необходимости туннель.

Нативные приложения с HTTP из кода часто **не упираются в CORS**; оболочки вроде Capacitor с `capacitor://` или `http://localhost` — добавьте в `CORS_ORIGINS` при необходимости.
