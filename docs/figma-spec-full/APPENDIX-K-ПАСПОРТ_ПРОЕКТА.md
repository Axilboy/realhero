# Паспорт проекта Real Hero

Документ фиксирует назначение, состав и правила сопровождения приложения **Real Hero** (рабочее имя пакета: `realhero`).

## 1. Общие сведения

| Поле | Значение |
|------|----------|
| **Наименование** | Real Hero |
| **Тип** | Веб-приложение (SPA + REST API) |
| **Назначение** | Личный кабинет: **финансы** (счета, операции, инвестиции, бюджет), **тело** (замеры, питание, тренировки), **задачи** (списки, сроки, задел под квесты), заготовка «Действия», точка входа «Герой» |
| **Репозиторий** | Монорепозиторий: фронтенд в корне, бэкенд в каталоге `api/` |
| **Текущая версия** | Фронт и API: semver в корневом `package.json` и `api/package.json`; см. `CHANGELOG.md` |
| **Лицензия** | Не указана в репозитории (при публикации задать явно) |

## 2. Целевая аудитория и контекст

- **Пользователь:** один владелец аккаунта (email + пароль), данные изолированы по `userId`.
- **Сценарий:** вход с браузера (десктоп / мобильный), работа в основном офлайн-устойчивого UI с запросами к своему серверу.
- **Продакшен:** развёртывание на Linux за nginx; статика фронта и Node API за обратным прокси (см. корневой `README.md` и `scripts/`).

## 3. Архитектура и технологии

### 3.1. Фронтенд (корень репозитория)

- **Стек:** React 18, TypeScript, Vite 6, React Router 7.
- **Сборка:** `npm run build` → каталог `dist/`.
- **Dev:** Vite dev-server, прокси `/api` → бэкенд (по умолчанию `127.0.0.1:3000`).
- **Аутентификация:** httpOnly cookie `rh_session` + при необходимости JWT в ответе (`Authorization` / локальное хранение токена для обхода ограничений прокси).

### 3.2. Бэкенд (`api/`)

- **Стек:** Fastify 5, Prisma ORM, SQLite (файл БД задаётся в `DATABASE_URL`).
- **Префиксы API:** `/api/v1/finance/*`, `/api/v1/body/*`, `/api/v1/tasks`, `/api/v1/hero`, плюс `/api/v1/auth/*` и `/health`.
- **Безопасность:** bcrypt для паролей, JWT для сессии, CORS по `CORS_ORIGINS`.
- **Сборка:** `npm run build` → `dist/`, запуск `node dist/index.js`.

### 3.3. Внешние сервисы (опционально)

- **MOEX ISS API** — поиск и котировки российских бумаг, оценка купона и дивидендов для подсказок в портфеле.
- **CoinGecko** — поиск и цены криптоактивов в рублёвом эквиваленте.

Частота запросов к внешним API ограничена кешированием на стороне сервера (~45 с).

## 5. Данные и модель

### 5.1. Пользователь (общее)

- Email, пароль.
- **Герой:** `heroTotalExp` (целое ≥ 0) — опыт для уровня; API `GET /api/v1/hero`, `POST /api/v1/hero/sync-from-local` (перенос с клиента без понижения сервера).
- **Финансы:** `financeReportingGranularity` (день / неделя / месяц / год / свой диапазон), `financeReportingDay` (1–28) для режима «месяц», опционально `financeReportingCustomFrom` / `financeReportingCustomTo`; флаги отображения доходности на карточках счёта (`financeCardShowYield*`).
- **Тело:** `bodyMassUnit` (KG | LB), `bodyLengthUnit` (CM | IN); цели питания: `bodyKcalGoal`, `bodyProteinGoalG`, `bodyFatGoalG`, `bodyCarbGoalG`.

### 5.2. Финансы

- **Счета:** типы вклад, накопительный, карты, счёт `BANK` с процентом на остаток и т.д.; оценка дохода по %% (в т.ч. день / неделя / месяц / год) — см. `accountInterest` и маршруты счетов.
- Категории операций (в т.ч. иерархия, эмодзи, цвет, архив); транзакции; переводы.
- **Удаление счёта:** `merge-into` (перенос на другой счёт) или `purge` (полное удаление истории).
- **Инвестиции:** позиции, котировки MOEX/CoinGecko, доход с бумаги; обзор портфеля исключает из блока «вклады с %%» кредитные карты и банковские счета с отрицательным балансом (долг).
- **Бюджет:** лимиты по категориям за календарный месяц (`CategoryBudget`).

### 5.3. Тело

- **BodyMeasurement** — дата (локальная строка YYYY-MM-DD), вес/рост/% жира/обхваты в **кг и см** в БД; заметка.
- **NutritionEntry** — день, приём пищи, строка блюда, ккал, БЖУ.
- **TrainingProgram** → **TrainingDay** (индекс в цикле) → **TrainingExercise** (вид: сила, кардио, растяжка, разминка, другое; подходы, вес, время, дистанция и т.д.).
- **WorkoutLog** + **WorkoutLogLine** — факт тренировки; в строке — `resultJson` (серии, кардио и пр.).

- **UserTask** — модуль «Задачи»: заголовок, заметка, локальный срок `dueDate` (YYYY-MM-DD) и опционально `dueTime` (HH:mm), `completedAt`, `sortOrder`, `source` (MANUAL \| QUEST), опционально `questInstanceId` / `questStepId` для будущих квестов (`ТЗ_КВЕСТЫ_И_ТУДУ.md`).

БД по умолчанию SQLite; для прода возможен PostgreSQL (см. корневой `docker-compose.yml`, `api/README.md`, переменная `DATABASE_URL`, смена `provider` в `schema.prisma`).

### 5.4. Интерфейс и зоны ответственности экранов

- **Главный экран финансов:** карточки счетов (карусель); плитки отчёта за период (доходы, расходы и переводы, баланс); при положительном пассивном потоке — **карусель «На что тянет пассивный доход»** (три слайда: день / неделя / месяц) с **примерами покупок** на соответствующие суммы (ориентиры по `src/lib/financeMotivationCards.ts`); затем сводка «На счетах», инвестиции, всего, оценка пассивного потока (~в месяц), разбивка капитала; при настроенном бюджете — полоса «осталось по лимитам».
- **Вкладки финансов (4):** главная, инвестиции, аналитика, бюджет — вертикальный свайп + нижняя поднавигация.
- **Инвестиции:** оценка бумаг, структура долей, метрики; карусель вкладов/накопительных (как на главной); **подсказки по пассивному доходу** — три блока текста (день / неделя / месяц), не карусель; без учёта кредиток и долговых счетов в блоке вкладов с %%.
- **Тело:** нижняя поднавигация модуля — **Главная** (сводка: вес, график, талия/тренировки, блок БЖУ за день, переход к замерам), **Питание**, **Тренировки**; экран **замеров** открывается с главной тела (не отдельная вкладка внизу). Шестерёнка — настройки единиц и целей КБЖУ (`body.*`, i18n).
- **Герой** — хаб: уровень/EXP (**`heroTotalExp`**, кэш в `localStorage`), мини-карточки **Тело** (ккал и строка БЖУ) и **Финансы** (всего по счетам и инвестициям, при ненулевом пассиве — оценка ~в месяц), блок **квестов** (активные и доступные), ряд **«Скоро»**. Блоки «Сегодня» (кнопки Тело/Задачи) и серии стриков на главной убраны для снижения перегруза. Полные отчёты — на вкладках модулей. **Задачи** — «Сегодня» / «Входящие» / «Все», поднавигация с иконками, синхронизация с API. **Действия** — заготовка UI; ТЗ: **`ТЗ_ДЕЙСТВИЯ.md`**; геймификация — п. 5.5.
- **Оболочка:** нижняя навигация приложения (5 разделов) с иконками и подписями; основной UI финансов и часть экранов на ключах i18n `fin.*` / `hub.*` / `body.*` / `todo.*` (ru/en).

### 5.5. Геймификация и виртуальный аватар (задел на будущее)

Эти направления **не обязаны** быть реализованы в текущей версии, но их нужно учитывать в архитектуре данных и UI, чтобы потом не ломать модуль «Тело» и экран «Герой».

- **Геймификация** — начисление очков, уровни, серии (streak), достижения, визуальная обратная связь за действия из реальной жизни: регулярные замеры и тренировки, соблюдение целей КБЖУ, завершённые тренировки, осмысленные финансовые привычки (без превращения денег в «очки ради очков»). Источником правды должны оставаться **уже существующие сущности** (`BodyMeasurement`, `NutritionEntry`, `WorkoutLog`, транзакции/бюджет — по решению продукта), а баллы и бейджи — **производные** (отдельные таблицы или вычисление по правилам), чтобы не дублировать ввод данных.

- **Виртуальный аватар** — персонаж на экране «Герой», визуально отражающий прогресс пользователя. Связь с модулем «Тело»: **антропометрия** (вес, рост, % жира, обхваты из `BodyMeasurement`) как параметры для масштаба/пропорций или стадий «формы»; **активность** (частота тренировок, завершённые `WorkoutLog`) — анимации состояния, экипировка, фоны. Контракт с бэкендом на будущее: хранить **снимок параметров аватара** (или ссылку на пресет + числовые множители), не привязывать жёстко к одному 3D-движку в первой итерации.

- **Интеграция в интерфейс:** мини-виджеты прогресса (полосы, «серия дней») логично подводят к экрану «Герой»; глубокие сцены аватара — не перегружать экраны ввода в модуле «Тело» (сводка, питание, тренировки, замеры), чтобы ввод оставался быстрым.

- **Квесты и модуль «Задачи»** — сценарные цепочки с шагами и наградами; исполняемые шаги **подтягиваются в туду** как связанные задачи, чтобы прогресс квеста и отметки в списке задач не расходились. Проектное ТЗ: **`ТЗ_КВЕСТЫ_И_ТУДУ.md`**.

## 6. Окружения

| Окружение | Назначение |
|-----------|------------|
| **Локальное** | `npm run dev` (фронт) + `npm run dev` в `api/` после `prisma db push` |
| **Продакшен** | nginx + статика + PM2/systemd для API; переменные из `api/.env` |

Обязательные для прода настройки: сильный `JWT_SECRET`, `CORS_ORIGINS`, при необходимости `COOKIE_DOMAIN` / `COOKIE_SECURE` (см. `api/README.md`).

## 7. Ограничения и риски

- Регистрация по email **без** подтверждения почты — не использовать как публичный сервис без доработок.
- Котировки и оценки купонов/дивидендов **ориентировочные**, не являются инвестиционной рекомендацией.
- На Windows пути в OneDrive иногда дают **EPERM** при `prisma generate` — при необходимости выполнять генерацию клиента вне синхронизируемой папки или после закрытия процессов, держащих DLL.
