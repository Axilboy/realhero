# ТЗ: фаза визуальной доводки Real Hero

**Продукт:** Real Hero (`realhero`) — SPA + REST API, монорепозиторий (фронт в корне, `api/`).  
**Версия документа:** 1.0  
**Дата:** 2026-03-30  
**Статус:** действующее ТЗ **текущей фазы разработки** — приоритетно для решений «что менять в UI сейчас».

---

## 1. Зачем этот документ

Свести в **одну систему координат**:

- **Паспорт** и доменные ТЗ — что продукт *умеет* и что нельзя сломать.
- **`ТЗ_UI_МОКАПЫ_V2.md`** и PNG в `docs/screenshots/v2/` — *как должно выглядеть* после входа.
- **`docs/figma-spec-full/`** и **`docs/FIGMA_AI_TZ.md`** — детализация для Figma / AI; местами устарели относительно согласованного UI v2.
- Экспорт **Figma Make** (`fmdownload/`, при наличии) — **референс визуала**, не отдельная кодовая база.

**Фокус фазы:** **допилить визуал** (отступы, типографика, состояния, согласованность с мокапами, i18n-длины) **без удаления и без упрощения уже внедрённого функционала**, кроме явно согласованных правок верстки.

## 3. Замороженный функциональный объём (не урезать в визуальной фазе)

Ниже — **что уже является частью продукта** и должно **сохраняться** (логика, API, сценарии). Меняется только представление: CSS, композиция, копирайт в рамках i18n, доступность.

### 3.1. Оболочка и навигация

- Один `AppShell` после входа: **карусель из 5 разделов** без смены URL между ними.
- Порядок вкладок: **Герой (0) → Тело (1) → Развитие / задачи (2) → Финансы (3) → Настройки (4)** — см. `src/lib/shellTabs.ts`.
- Шапка: для «Герой» — двухрядный вариант (заголовок, шестерёнка, колокол-заглушка; второй ряд — email, RU|EN, выход); для остальных — email + язык + выход.
- Нижняя навигация: иконки + подписи, активное состояние.

### 3.2. Аутентификация и гость

- Лендинг, вход, регистрация, защита `/app/*` — как в `ПАСПОРТ_ПРОЕКТА.md` §3.

### 3.3. Модуль «Герой»

- Уровень, EXP, синхронизация с `GET/POST /api/v1/hero`, кэш в `localStorage`.
- **Без** блоков «Сегодня» и **без** стриков на главной (паспорт, ТЗ v2).
- Мини-карточки **Тело** (ккал, БЖУ) и **Финансы** (всего, пассив при >0).
- Квесты: активные и доступные, кнопки «К задачам», «Отказаться», «Взять» — по текущему API квестов.
- Блок «Скоро» (заглушки).

### 3.4. Модуль «Финансы»

- Четыре подвкладки: главная, инвестиции, аналитика, бюджет; вертикальный свайп.
- Счета, операции, переводы, категории (дерево, модалка по ТЗ v2-06), инвестиции, MOEX/CoinGecko, мотивация по пассиву, бюджет, отчётный период, удаление счёта merge/purge — **как в коде и паспорте**.

### 3.5. Модуль «Тело»

- Три подвкладки: главная, питание, тренировки; замеры — отдельный поток с главной, форма «дата + вес», доп. поля в `<details>` — **как в ТЗ v2 и паспорте**.

### 3.6. Модуль «Задачи» (Развитие)

- Вкладки Сегодня / Входящие / Все, синхронизация с `/api/v1/tasks`, задел под квесты в данных.

### 3.7. «Действия» / настройки

- Экран настроек (карточная раскладка), заготовка «Действия» — **сохраняются**; полноценная логика «Действия» и расширение квестов — по отдельным ТЗ (`ТЗ_ДЕЙСТВИЯ.md`, `ТЗ_КВЕСТЫ_И_ТУДУ.md`), **не цель этой визуальной фазы**.

### 3.8. Бэкенд и внешние API

- Fastify, Prisma, SQLite/Postgres, эндпоинты из паспорта; кеш внешних котировок ~45 с — **без изменений в рамках «только визуал»**, кроме багфиксов, блокирующих отображение данных.

## 4. Цели визуальной фазы (что делаем сейчас)

1. **Сверка с эталонами** `docs/screenshots/v2/v2-rh-*.png` (при появлении в репо): ширина 428px, токены `--rh-*`, ритм отступов, читаемость чисел.
2. **Единообразие:** поднавигации модулей, модалки, FAB, типографика секций (uppercase/overline где в ТЗ).
3. **Состояния:** loading / empty / error — не ломать вёрстку (скелетоны, короткие тексты).
4. **i18n RU|EN:** не обрезать критичный текст; при необходимости — уменьшение кегля или перенос, **без** удаления ключей.
5. **Доступность:** контраст, `focus-visible`, `aria` для интерактива (в пределах текущих компонентов).
6. **Документация:** при изменении экрана — обновлять **`ТЗ_UI_МОКАПЫ_V2.md`** чеклисты и при необходимости **`docs/figma-spec-full/PART-03`** (Герой) так, чтобы **не противоречили** паспорту.

## 5. Дизайн-система (не менять смысл токенов без причины)

- Источник: **`src/index.css`**, `:root` — `--rh-bg`, `--rh-surface`, `--rh-accent`, `--rh-shell-max: 428px`, и т.д. (см. **`docs/figma-spec-full/PART-01-design-system.md`**).
- Любая смена токена — **осознанно** и **сквозь все экраны**, с пометкой в `CHANGELOG.md`.

| Артефакт | Роль в проекте |

| **`docs/figma-spec-full/`** | Подробные чеклисты; синхронизировать с п. 2 при расхождениях. |

---

## 7. Вне объёма визуальной фазы

- Новые эндпоинты и сущности БД «ради красоты».
- Переписывание стека (не Vite/React, не отказ от Prisma) без отдельного решения.
- Реализация **полного** сценария квестов/«Действий» из проектных ТЗ — только если это отдельно запланировано; **визуальная фаза** может **подготовить места** в UI, но не обязана закрывать логику.
