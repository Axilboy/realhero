# Паспорт проекта Real Hero

Документ фиксирует назначение, состав и правила сопровождения приложения **Real Hero** (рабочее имя пакета: `realhero`).

## 1. Общие сведения

| Поле | Значение |
|------|----------|
| **Наименование** | Real Hero |
| **Тип** | Веб-приложение (SPA + REST API) |
| **Назначение** | Личный кабинет: финансы (счета, операции, инвестиции), заготовки под разделы «Тело», «Задачи», «Действия», точка входа «Герой» |
| **Репозиторий** | Монорепозиторий: фронтенд в корне, бэкенд в каталоге `api/` |
| **Лицензия** | Не указана в репозитории (при публикации задать явно) |

## 2. Целевая аудитория и контекст

- **Пользователь:** один владелец аккаунта (email + пароль), данные изолированы по `userId`.
- **Сценарий:** вход с браузера (десктоп / мобильный), работа в основном офлайн-устойчивого UI с запросами к своему серверу.
- **Продакшен:** развёртывание на Linux за nginx; статика фронта и Node API за обратным прокси (см. корневой `README.md` и `scripts/`).

## 3. Архитектура и технологии

### 3.1. Фронтенд (корень репозитория)

- **Стек:** React 18, TypeScript, Vite 6, React Router 7.
- **Сборка:** `npm run build` → каталог `dist/`.
- **Dev:** Vite dev-server, прокси `/api` → бэкенд (по умолчанию `127.0.0.1:3000`).
- **Аутентификация:** httpOnly cookie `rh_session` + при необходимости JWT в ответе (`Authorization` / локальное хранение токена для обхода ограничений прокси).

### 3.2. Бэкенд (`api/`)

- **Стек:** Fastify 5, Prisma ORM, SQLite (файл БД задаётся в `DATABASE_URL`).
- **Безопасность:** bcrypt для паролей, JWT для сессии, CORS по `CORS_ORIGINS`.
- **Сборка:** `npm run build` → `dist/`, запуск `node dist/index.js`.

### 3.3. Внешние сервисы (опционально)

- **MOEX ISS API** — поиск и котировки российских бумаг, оценка купона и дивидендов для подсказок в портфеле.
- **CoinGecko** — поиск и цены криптоактивов в рублёвом эквиваленте.

Частота запросов к внешним API ограничена кешированием на стороне сервера (~45 с).

## 5. Данные и модель

- **Пользователь:** email, пароль, поле **`financeReportingDay`** (1–28) — с какого числа каждого месяца начинается **отчётный период** для сводки на главном экране и для прогноза в аналитике.
- **Счета:** типы включая вклад (`DEPOSIT`) и накопительный (`SAVINGS`) с опциональной **годовой ставкой %**; оценка дохода в месяц по ставке.
- Категории операций, транзакции (расход/доход), переводы между счетами.
- **Удаление счёта при наличии операций и переводов:** перенос всех связанных записей на выбранный счёт (`merge-into`), затем удаление; петлевые переводы «счёт → тот же счёт» очищаются.
- **Инвестиции:** вид актива, количество, цена, котировки MOEX/CoinGecko, доход с бумаги за год или за месяц (в БД — годовой эквивалент).
- БД по умолчанию SQLite; при росте нагрузки возможен переход на PostgreSQL в Prisma.

### 5.1. Интерфейс и зоны ответственности экранов

- **Главный экран финансов:** итоги «на счетах / инвестиции / всего», оценка **пассивного дохода ~в месяц** (вклады по ставке + бумаги по введённым купонам/дивидендам), **разбивка капитала** (вклады, накопительные, акции, облигации, прочие инструменты; отдельно сумма вне этой структуры — карты, наличные, обычные счета), **текущий отчётный период** (доходы/расходы/баланс по периоду), настройка отчётного числа.
- **Вклад «Инвестиции»:** оценка только **бумаг**, структура долей без карт и без строки «капитал всего»; вклады/накопительные — каруселью; обновление данных после сделок синхронизируется с главным экраном.
- **Аналитика:** блок **прогноза бюджета к концу отчётного периода** (среднедневные доход/расход по факту периода, линейная экстраполяция); ниже — разбивка по категориям за **календарный месяц** (отдельный выбор месяца).

## 6. Окружения

| Окружение | Назначение |
|-----------|------------|
| **Локальное** | `npm run dev` (фронт) + `npm run dev` в `api/` после `prisma db push` |
| **Продакшен** | nginx + статика + PM2/systemd для API; переменные из `api/.env` |

Обязательные для прода настройки: сильный `JWT_SECRET`, `CORS_ORIGINS`, при необходимости `COOKIE_DOMAIN` / `COOKIE_SECURE` (см. `api/README.md`).

## 7. Ограничения и риски

- Регистрация по email **без** подтверждения почты — не использовать как публичный сервис без доработок.
- Котировки и оценки купонов/дивидендов **ориентировочные**, не являются инвестиционной рекомендацией.
- На Windows пути в OneDrive иногда дают **EPERM** при `prisma generate` — при необходимости выполнять генерацию клиента вне синхронизируемой папки или после закрытия процессов, держащих DLL.
